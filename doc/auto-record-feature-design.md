# 自動記録機能 設計書 v0.1

## 0. 位置づけと責務

### 前提となるアプリのレイヤー構造

```
[コア] 記録 = 緯度経度 + 時間
   ├─ 逆ジオコーディング → 地名表記（表示ラベル）
   ├─ 座標 → 近隣スポット自動表示 → 場所名の簡単入力
   └─ ラベル / メンバーの任意設定
        ↓ この上に乗る
[判定レイヤー] 聖地巡礼機能
   └─ 記録の座標がスポット判定距離内 → 「聖地巡礼達成」
```

### 自動記録機能の責務（スコープ）

> **信頼できる Visit（緯度経度＋時間）を候補として貯め、ユーザーが起動時に承認したら「通常の記録」として確定させる。**

- 確定後は **手動記録とまったく同一のパイプライン**（逆ジオ → 近隣スポット表示 → ラベル/メンバー → コース判定）に流れる
- 記録後の処理は **一切再発明しない**。自動記録は「記録の入口を増やすだけ」
- ブルーバッジの定義 = **位置データの出所がシステムか否か**。Visit は OS 由来なのでブルー資格を持つ（手入力のオレンジと対比）

### スコープ外（やらないこと）

- 自動記録専用の地名表示ロジック（既存の逆ジオを使う）
- 自動記録専用のスポット紐付け・コース判定ロジック（確定後に既存処理が走る）
- リアルタイム性（Visit は数分〜十数分遅延して確定する。後で見返す用途なので許容）

## 1. 技術的土台: Core Location Visit Monitoring

### 採用 API

`CLLocationManager.startMonitoringVisits()` / `CLVisit`

| 特性 | 内容 |
|------|------|
| 消費電力 | 極小。OS が他アプリと共有するセル/WiFi/加速度由来の位置を再利用 |
| バックグラウンド | アプリ終了後も OS が Visit を検知し、必要時にアプリを起こしてコールバック |
| 取得データ | `coordinate` / `arrivalDate` / `departureDate` / `horizontalAccuracy` |
| 権限 | **`Always`（常に許可）が必須**。`When In Use` ではバックグラウンド監視不可 |

### CLVisit の制約（設計で吸収する）

1. **精度が粗い**: `horizontalAccuracy` が数十〜数百m。→ 足切りで対応
2. **遅延がある**: 出発確定後にコールバック。→ 後で見返す用途なので問題なし
3. **検知漏れ/誤検知**: 短時間滞在を拾わない / 信号待ちを拾う。→ 滞在時間足切りで対応
4. **`departureDate` が未確定の場合がある**: 到着のみ先行通知され `departureDate == nil` のことがある。→ 滞在時間が計算できない暫定 Visit の扱いを定義（後述）

## 2. 足切りロジック（候補採用条件）

自動記録機能の唯一の作り込みポイント。**「Visit の緯度経度そのものの信頼性」だけ**で判断する（スポットとの距離やコースは一切見ない）。

### 採用条件（すべて満たす Visit のみ候補化）

| 条件 | 初期値（チューニング可能） | 目的 |
|------|------|------|
| 精度足切り | `horizontalAccuracy <= 200m` | 座標がブレすぎた Visit を捨てる |
| 滞在時間足切り | `departureDate - arrivalDate >= 5分` | 信号待ち等の通過を捨てる |
| `departureDate == nil` の扱い | 候補化を保留（次回コールバックで再評価） | 滞在時間が判定できないため |

- これらの数値はすべて **Remote Config / 定数で外出し** し、後からチューニング可能にする
- 足切りで落ちた Visit は **保存しない**（ユーザーに見せないノイズを溜めない）

### 補足: なぜスポット距離で足切りしないか

記録のコアは「緯度経度＋時間」。スポット紐付けもコース判定も確定後の既存処理。よって候補化段階では純粋に座標の信頼性だけを見る。近くにスポットが無い場所（自宅・コンビニ等）の Visit も、精度さえ満たせば候補になる（＝汎用の訪問ログ）。これは仕様通り。

## 3. データモデル

### VisitCandidate（候補エンティティ・CoreData）

承認前の一時データ。承認 or 却下されたら削除する。

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | UUID | 候補の一意識別子 |
| `latitude` | Double | Visit の緯度 |
| `longitude` | Double | Visit の経度 |
| `arrivalDate` | Date | 到着時刻 |
| `departureDate` | Date? | 出発時刻（暫定時は nil） |
| `horizontalAccuracy` | Double | 取得時の精度（m） |
| `placeName` | String? | 逆ジオ結果のキャッシュ（遅延取得・後述） |
| `status` | String | `pending` / `dismissed`（承認時は本記録へ移して削除） |
| `detectedAt` | Date | コールバックで受信した日時 |

### 確定後

VisitCandidate は **新しいテーブルを持たない**。承認されたら既存の記録エンティティ（手動記録と同じ）に変換して挿入し、VisitCandidate からは削除。確定後は既存パイプラインが走る。

```
VisitCandidate (pending)
   │ ユーザーが承認
   ▼
既存の記録エンティティ（手動と同型・ブルー扱い）
   │ 既存処理が自動適用
   ▼
逆ジオ / 近隣スポット表示 / ラベル・メンバー / コース判定
```

## 4. 処理フロー

### 4-1. バックグラウンド: Visit 受信

```
OS が Visit を検知
   ▼
locationManager(_:didVisit:) コールバック
   ▼
足切り判定（精度・滞在時間・departureDate有無）
   ├─ 不採用 → 破棄（何も保存しない）
   └─ 採用 → VisitCandidate を status=pending で保存
              （※この時点では逆ジオしない。バックグラウンド処理を最小化）
```

### 4-2. フォアグラウンド: 起動時レビュー

```
アプリ起動 / フォアグラウンド復帰
   ▼
pending な VisitCandidate を取得
   ▼
未取得の placeName を逆ジオcoding でまとめて補完（表示直前に実行）
   ▼
「未確認の訪問候補が N件あります」を提示
   ▼
ユーザーが候補ごとに
   ├─ 承認 → 記録として確定（→ 既存パイプライン）
   └─ 却下 → VisitCandidate を削除
```

### 設計判断: 逆ジオは「表示直前」に行う

- バックグラウンドで逆ジオすると通信・電力・API コストが乗る上、ユーザーが見ない候補にも無駄に走る
- フォアグラウンドで表示する候補だけ逆ジオすれば最小コスト
- 逆ジオ失敗時は座標のみ表示でフォールバック（地名はあくまで補助ラベル）

## 5. UI 設計（骨子）

### 5-1. 起動時の気づき

- 起動時、pending 候補があれば **控えめな通知導線**（バッジ/バナー）で「未確認の訪問が N件」
- 押し付けない。聖地巡礼の達成演出（青チェックの感動）とは別物の、静かな「溜まってますよ」表現

### 5-2. 候補レビュー画面

各候補カードに表示する情報:

- 地名（逆ジオ結果。失敗時は座標）
- 日時（arrivalDate ベース、滞在時間も併記可）
- ミニマップ（座標ピン）
- 近隣スポット候補（既存の近隣スポット表示を流用 → ワンタップで場所名入力）

アクション:

- **承認** → 記録確定。ここで既存の記録編集導線（ラベル・メンバー設定）に繋げられる
- **却下** → 候補削除
- **まとめて却下** → 古い候補の一括処理

### 5-3. 確定後

確定した記録は手動記録とまったく同じ見た目・扱い（ブルーバッジ）。自動由来か手動由来かを内部的に持つかは任意（分析用に `recordSource` を持たせる選択肢はあるが、UI 上は区別しなくてよい）。

## 6. 権限・オンボーディング・審査

### 6-1. 権限取得フロー

`Always` はいきなり要求すると拒否されやすい。段階的に:

```
1. 機能を使う文脈で「なぜ常時位置情報が必要か」を画面で説明
2. When In Use を先に取得（既存記録機能で使用）
3. 自動記録をオンにするタイミングで Always への昇格を要求
4. iOS が後日出す「Always を継続しますか」プロンプトに耐えられるよう、
   価値を実感させてから昇格させる
```

### 6-2. Info.plist

- `NSLocationAlwaysAndWhenInUseUsageDescription` に **具体的な利用目的** を記載
  - 例: 「訪れた場所を自動で記録候補として保存し、後で見返して記録できるようにするため、常に位置情報を使用します」
- `UIBackgroundModes` に `location`

### 6-3. App Store 審査対策

- `Always` 権限はレビューで利用目的を厳しく問われる
- **レビューノートに自動記録機能の目的とユーザーメリットを明記**
- 権限要求前に説明画面を挟む実装にしておく（審査官が確認できる）
- 設定で **自動記録を OFF にできる** ようにする（OFF 時は Always を使わない）

---

## 7. 設定・プライバシー

- 設定画面に「自動記録」トグル（デフォルト OFF が無難。ユーザーが理解して ON にする）
- pending 候補の保持上限/保持期間（例: 30日 or 最大100件で古いものから自動削除）
- 候補データは端末内のみ（既存のデータポリシー「データは端末内に保存」と整合）

## 8. 段階的実装プラン

| フェーズ | 内容 |
|---------|------|
| Phase 1 | Visit 監視 + 足切り + VisitCandidate 保存 + 起動時レビュー画面（承認→既存記録確定）。逆ジオは表示時。 |
| Phase 2 | 近隣スポット候補のレビュー画面への統合、まとめて操作、保持期間管理 |
| Phase 3 | 権限オンボーディングの作り込み、Remote Config による足切り値チューニング |
| Phase 4 | 分析（`recordSource` 集計で自動記録の承認率を見てチューニング） |

## 9. 未決事項 / 要検討

1. 足切り初期値（200m / 5分）の妥当性 — 実機でログを取って調整
2. `departureDate == nil` 暫定 Visit を「どれだけ待って確定させるか」
3. 自動記録トグルのデフォルト ON/OFF（審査・UX・電力のバランス）
4. 候補の保持上限・期間の具体値
5. 起動時レビューの通知強度（うるさくない範囲）
